# RTK GPS 처음 시작하기 — 시행착오 일기

> WTRTK-982 모듈로 한국 NGII NTRIP을 연결하면서 배운 것들.
> 비유 중심으로 풀어쓴 입문자 문서.

---

## 1. RTK가 도대체 뭐길래?

### 일반 GPS의 한계

스마트폰 GPS를 쓰면 위치가 보통 **2~5 m** 오차가 납니다. 길 안내에는 충분하지만, "자율주행 차량이 차선을 지키도록 한다"거나 "드론이 정확히 농약을 뿌린다" 같은 일에는 턱없이 부족합니다.

이건 GPS의 본질적 한계인데, 이유는:
- 위성에서 오는 신호가 **대기권(이온층/대류층)을 통과하면서 미세하게 휘어짐**
- 위성 시계와 수신기 시계가 완전히 일치하지 않음
- 위성 궤도 데이터가 약간 부정확함

### RTK라는 발상

"근처에 **정확한 위치를 미리 아는 기준국**을 두자. 기준국이 'GPS는 나를 지금 X로 알려주는데, 진짜 내 위치는 Y야. 따라서 오차는 Y-X'라고 알려주면, 그 오차를 너도 빼주면 되잖아?"

이게 RTK(Real-Time Kinematic)입니다.

**비유**: 시험을 본다고 생각해보세요.
- 일반 GPS = 혼자 시험 보기 (점수 60점 ± 20점)
- RTK = 옆에 정답을 아는 친구가 "1번은 답이 3이래"라고 실시간으로 알려줌 (점수 99점 ± 1점)

이 "정답을 아는 친구"가 **기준국(Base Station)**이고, 그 친구가 보내주는 정보가 **RTCM 보정 데이터**입니다.

## 2. 우리 시스템의 구성 요소

```
[하늘의 위성들]                              [기준국 (NGII 운영)]
     │ 신호                                       │ 보정 정보
     ▼                                            ▼
[안테나] ─── USB ─── [컴퓨터 (main.py)] ─── 인터넷 ─── [NTRIP 서버]
     │                     │
     ▼                     ▼
[RTK 모듈              (RTCM 받아서
 WTRTK-982]            모듈에 전달)
```

### 각 부품의 역할

| 부품 | 역할 | 비유 |
|------|------|------|
| **안테나** | 위성 신호 수신 | 귀 |
| **RTK 모듈 (WTRTK-982)** | 신호 해석, 위치 계산 | 뇌 |
| **컴퓨터 (main.py)** | 인터넷에서 보정 받아서 모듈에 전달 | 우체부 |
| **NTRIP 서버 (NGII)** | 기준국 정보 중계 | 기준국 친구네 집 |
| **RTCM** | 보정 데이터 포맷 | 친구가 쓰는 언어 |
| **NMEA** | 모듈이 컴퓨터에 출력하는 결과 포맷 | 뇌가 보고하는 형식 |

## 3. 데이터 흐름 — 처음 켰을 때 무슨 일이 일어나나

### Step 1: 모듈 혼자 시작 (NTRIP 접속 전)

전원만 켜면 모듈이 위성 신호만으로 대충 위치를 잡습니다. 이때 모듈은 NMEA 문장을 시리얼 포트로 1Hz(초당 1번) 토해냅니다.

예시:
```
$GPGGA,101805.00,3728.67011,N,12653.75725,E,1,08,1.0,53.4,M,...
                                          ↑
                              이 숫자가 핵심! (Fix Quality)
```

7번째 필드(`1`)가 **현재 RTK 등급**입니다:
- `0` = 위치 못 잡음
- `1` = 일반 GPS (2~5m 정밀도)
- `2` = DGPS (0.5~2m)
- `4` = RTK Fixed (1~2cm) ← 우리의 목표
- `5` = RTK Float (10~50cm)

### Step 2: 컴퓨터가 모듈의 현재 위치를 읽음

`main.py`가 모듈이 출력하는 `$GPGGA` 또는 `$GNGGA` 문장을 기다립니다. 위치가 잡혔다는 신호(`1` 이상)가 오면 다음 단계로.

**왜 위치를 먼저 알아야 할까?**
NTRIP 서버는 "네 근처에 있는 가상 기준국을 만들어 줄게"라고 동작합니다(VRS = Virtual Reference Station). 그래서 서버에게 **"나 여기 있어"**라고 알려줘야 적합한 보정 데이터를 골라서 보내줍니다.

**비유**: 음식 배달 시키려면 주소를 알려줘야 가장 가까운 지점에서 출발하잖아요. 그것과 같습니다.

### Step 3: NTRIP 서버에 HTTP 비슷한 접속

```python
GET /VRS-RTCM34 HTTP/1.0
Authorization: Basic <ID:비밀번호 Base64>
```

마운트포인트(`VRS-RTCM34`)는 "어떤 종류의 보정을 받을지" 선택하는 메뉴 같은 것. NGII는 여러 메뉴를 제공합니다:
- `VRS-RTCM31`: 오래된 포맷, GPS+GLONASS만
- `VRS-RTCM34`: 최신 포맷, GPS+GLONASS+Galileo+BeiDou+QZSS **(우리가 쓰는 것)**

서버가 `ICY 200 OK`로 답하면 성공.

### Step 4: GGA 전송 + RTCM 받기 시작

내 위치를 한 번 보내주면(`ntrip.send(strGNGGA.encode())`), 서버가 그 위치 근처의 보정 데이터를 **계속 흘려보내기 시작**합니다 (스트리밍).

### Step 5: 받은 RTCM을 모듈에 그대로 전달

```python
rtcm = ntrip.recv(4096)
RTK.write(rtcm)
```

컴퓨터는 그냥 우체부 역할. 내용물(RTCM) 해석은 모듈이 합니다. RTCM은 **바이너리**(사람이 읽을 수 없는 0xD3 시작하는 바이트 덩어리)라 화면에 출력해도 글자가 안 보입니다.

### Step 6: 30초마다 GGA 재전송 (연결 유지)

서버에 "나 아직 살아있고, 위치는 여전히 여기야"라고 신호. 안 하면 서버가 연결 끊습니다.

## 4. NMEA 문장 읽는 법

모듈이 토해내는 `$로 시작하는 한 줄`들을 NMEA 문장이라고 합니다. 첫 5글자가 **누가 만든 정보인지** 알려줍니다.

### Talker ID — "어느 위성 가족이 만들었나"

| 접두사 | 의미 | 국가 |
|--------|------|------|
| `$GP` | **G**PS | 미국 |
| `$GL` | G**L**ONASS | 러시아 |
| `$GA` | G**A**lileo | EU |
| `$GB` | Bei**D**ou | 중국 |
| `$GQ` | **Q**ZSS | 일본 |
| `$GN` | **N**et (모든 위성 가족 통합) | — |

**비유**: 학교 보고서 같은 거. `$GP`는 "GPS팀이 작성한 보고서", `$GN`은 "모든 팀이 협력해서 작성한 종합 보고서".

### 문장 종류

| 종류 | 내용 | 예시 의미 |
|------|------|-----------|
| `GGA` | 시간, 위치, 고도, RTK 등급 | "지금 N37° 28' 위치, 53m 높이, RTK Fixed" |
| `RMC` | 위치, 속도, 방향 | "여기, 시속 0.5km, 동쪽으로" |
| `GSA` | 어떤 위성 썼는지, DOP | "PRN 10, 12, 24번 위성 사용, HDOP 0.8" |
| `GSV` | 보이는 위성 신호 강도 | "PRN 10번 위성, 신호 강도 41dB" |
| `VTG` | 속도/방향 | "동쪽 시속 0.5km" |

### 예시 해독

```
$GNGGA,123137.00,3728.66743,N,12653.75440,E,5,29,0.6,49.4,M,...
       ↑        ↑          ↑              ↑ ↑  ↑
       시간     위도        경도           │ │  HDOP (낮을수록 좋음)
                                          │ 사용 위성 수
                                          Fix 등급 (5=Float)
```

이 한 줄에서 우리가 알 수 있는 것:
- 12시 31분 37초
- 위도 37° 28.66743'N
- 경도 126° 53.75440'E
- **RTK Float 상태**, 29개 위성 사용, HDOP 0.6 (매우 좋음)

## 5. RTCM 메시지 — 기준국이 보내주는 보정 정보

RTCM은 바이너리지만, 0xD3로 시작하는 헤더 뒤에 **메시지 타입(12비트)**이 있습니다. 우리가 받은 타입들:

| 타입 | 내용 | 비유 |
|------|------|------|
| **1005** | 기준국의 정확한 좌표 | "내 집은 정확히 여기야" |
| **1007/1033** | 기준국 안테나/수신기 정보 | "내가 쓰는 장비는 ○○야" |
| **1075** | GPS 관측 (MSM5) | "GPS 위성 신호 측정한 값" |
| **1085** | GLONASS 관측 | "GLONASS 위성 신호 측정한 값" |
| **1095** | Galileo 관측 | "Galileo 위성 신호 측정한 값" |
| **1115** | QZSS 관측 | "QZSS 위성 신호 측정한 값" |
| **1304** | BeiDou 관측 | "BeiDou 위성 신호 측정한 값" |
| **1230** | GLONASS 위상 바이어스 | (위상 보정용 보조 정보) |
| **4094** | 제조사 전용 (Trimble) | (NGII 자체 메시지) |

**MSM**(Multiple Signal Message)이 최신 포맷, 그 외 1004/1012 같은 건 옛날 포맷.

## 6. RTK 등급의 본질 — 왜 Float에서 Fixed로 못 가나?

이게 RTK 핵심 개념입니다. 잘 이해하면 디버깅이 쉬워집니다.

### 코드(Code) 측정 vs 위상(Carrier Phase) 측정

GPS 위성에서 오는 신호는 두 가지 정보를 동시에 들고 옵니다:

**1) 코드(Code)** — 위성이 "지금 몇 시야"를 알려주는 디지털 신호. 받은 시각과 비교하면 거리가 나오는데, **해상도가 30cm짜리 자**로 재는 정도. → GPS, DGPS 정밀도.

**2) 반송파(Carrier Wave)** — 코드를 실어 나르는 전파 자체. **파장이 19cm**라서, 한 파장 안에서의 위치는 mm 단위로 잴 수 있음. → 이게 RTK의 핵심.

### 그런데 "몇 번째 파장이냐?"가 문제

비유: 운동장 트랙을 누가 몇 바퀴 돌았는지 모르고, 지금 어디쯤 있는지만 안다고 칩시다.

- "트랙 250m 지점에 있다" → 알 수 있음 (정밀)
- "그런데 몇 바퀴 돌고 250m 지점이야?" → ??? (정수 N)

이걸 **정수 모호성(Integer Ambiguity)**이라고 합니다. 위성과 수신기 사이 거리 = (정수 N × 19cm) + (정밀하게 잰 위상). 정수 N을 모르면 절대 거리를 못 정합니다.

### Float vs Fixed의 차이

| 상태 | N 값 | 정밀도 | 비유 |
|------|------|--------|------|
| **Float** | 실수로 추정 (예: 7234.6) | 10~50 cm | "한 7234바퀴 반쯤 돈 것 같아" |
| **Fixed** | 정수로 확정 (예: 7235) | 1~2 cm | "정확히 7235바퀴" |

Float는 N을 못 풀어서 실수로 어림한 상태. Fixed는 **여러 위성의 정수 N을 동시에 푸는 알고리즘(LAMBDA 등)**으로 정확히 정수를 찾은 상태.

### Fixed로 가려면 뭐가 필요?

1. **위성이 많아야** (10개 이상 권장) — 미지수가 많으니 방정식도 많이 필요
2. **신호가 깨끗해야** — 위상이 흔들리면 정수 찾기 실패
3. **시간이 충분해야** — 보통 수십 초

### 우리 데이터에서 본 증거

마지막 디버그 출력:
```
sats=29, HDOP=0.6, fix=RTK Float
```

위성 29개! HDOP 0.6 (이상적)! 그런데도 Float에 머무름. 이건 **2번 (신호 품질)** 문제입니다.

GSV 신호 강도를 보면:
```
일부 위성: SNR 50, 46, 43 (양호)
일부 위성: SNR 12, 17, 19 (나쁨)
```

높은 SNR 위성과 낮은 SNR 위성이 섞여 있는데, **낮은 SNR은 보통 다중경로(multipath)** — 건물·창문에서 반사된 가짜 신호.

## 7. 안테나 위치가 결정적인 이유

### 다중경로(Multipath)란?

위성에서 직접 오는 신호 + 근처 건물/유리/금속에서 **반사된 신호**가 동시에 안테나에 들어옴.

**비유**: 협곡에서 "야호" 외치면 메아리가 여러 번 옴. 어느 게 진짜 외친 순간인지 헷갈림.

코드 측정은 다중경로에 좀 강한데(GPS, DGPS는 영향 작음), **위상 측정은 다중경로에 매우 민감**합니다. 반사파 한 번이면 위상이 휙휙 바뀌어서 정수 모호성을 못 풉니다.

### 환경별 RTK 성공률

| 환경 | Float | Fixed |
|------|-------|-------|
| 실내 (창문 없음) | ❌ | ❌ |
| 실내 (창문 옆) | 가능 | 거의 불가능 |
| 발코니 | 쉬움 | 어려움 (가끔 잡힘) |
| 마당 (반쯤 트인 곳) | 쉬움 | 가능 (1~3분) |
| **옥상 / 야외 개활지** | **즉시** | **수 초~수십 초** |

지금 창문에서 Float만 잡히는 건 **이론적으로 정상**입니다.

## 8. 우리가 겪은 시행착오들

### 실수 1: `$GNGGA`만 받도록 코드 작성

처음 `main.py`는 `$GNGGA` 문장만 기다렸는데, 모듈은 기본 설정에서 `$GPGGA`를 출력하고 있어서 **무한정 대기**했습니다.

**교훈**: 모듈의 NMEA talker ID가 어떻게 설정돼있는지 먼저 확인. `$GN`(통합)이 아닌 `$GP`(GPS 전용)일 수도 있음.

### 실수 2: GPS+GLONASS만 받는 마운트포인트 사용

처음 `VRS-RTCM31`을 썼는데, NGII 소스테이블 확인 결과 이 마운트는 **GPS+GLONASS 2종**만 제공. Galileo/BeiDou 위성을 추적하지 못해서 가용 위성이 7개에 그쳤습니다.

`VRS-RTCM34`로 바꾸니 **5종 위성군 보정** 다 들어옴. (소스테이블 보는 법: `curl --http0.9 http://RTS1.ngii.go.kr:2101/`)

**교훈**: NTRIP 마운트포인트마다 지원 위성 시스템이 다름. 항상 소스테이블 먼저 확인.

### 실수 3: `MASK` 명령어 의미 착각

UM982에서 다중 GNSS를 "켜는" 명령어로 `MASK GPS`, `MASK GLO` 등을 보냈는데, **반대로 비활성화**되는 명령이었습니다.

```
MASK <system>   = 비활성화 (mask out)
UNMASK <system> = 활성화
```

이름이 헷갈리게 지어진 명령어. **응답이 `OK`로 나와도, 실제로 의도한 결과인지는 NMEA 출력으로 검증**해야 합니다.

저장(SAVECONFIG)까지 해버려서 영구화됐는데, `UNMASK`로 복구.

**교훈**:
1. 설정 명령어는 영구 저장 전에 결과 검증
2. 명령어 한 줄 보낸 뒤 NMEA가 어떻게 바뀌는지 봐야 함

### 실수 4: 존재하지 않는 NMEA 로그 이름 사용

`LOG GLGSV ONTIME 1`, `LOG GBGSV ONTIME 1` 같은 명령은 UM982가 거부(`PARSING FAILD GRAMMAR ERROR`).

이유: UM982의 NMEA 출력 체계에서는 **로그 이름은 통합형(`GPGSV`, `GNGSA` 등)으로만** 등록 가능. 다만 출력될 때는 talker ID(`$GP/$GL/$GA/$GB/$GQ`)가 위성군별로 자동 분리되어 나옴.

즉 `LOG GPGSV ONTIME 1` 하나만 켜면 모든 위성군의 GSV가 자동 출력됩니다.

**교훈**: 각 모듈/펌웨어마다 명령어 어휘가 다름. 매뉴얼을 봐야 함.

## 9. 정상 상태 체크리스트

지금 우리 시스템에서 검증된 정상 동작:

✅ NTRIP 서버 접속 (`ICY 200 OK`)
✅ RTCM 데이터 스트리밍 (초당 ~700 bytes)
✅ 5종 위성군 보정 수신 (1075/1085/1095/1115/1304)
✅ 모듈 다중 GNSS 추적 (29개 위성)
✅ HDOP 0.6 (이상적 위성 배치)
✅ RTK Float 도달

❌ RTK Fixed 미도달 (안테나 환경 문제)

남은 한 가지는 **안테나를 야외로**.

---

## 10. 다음에 시도해볼 것들

### A. 야외 테스트
옥상이나 마당으로 가서 30초~2분 두면 Fixed로 점프할 것. 이게 됐다면 시스템 자체는 완벽.

### B. 안테나 위치 최적화
Fixed가 잡히는 환경을 찾았다면:
- 안테나 주변에 금속·유리 없는 곳
- 하늘이 270° 이상 트인 곳
- 안정적으로 고정 (흔들리면 위상 측정에 영향)

### C. 듀얼 안테나 모드 (WTRTK-982 특화 기능)
이 모듈은 안테나 2개로 **방위각(heading)도 측정** 가능. 차량/드론에 쓸 때 매우 유용.
- `CONFIG HEADING LENGTH` 명령으로 안테나 간격 설정 필요
- 두 안테나가 모두 Fixed여야 정확한 헤딩 나옴

### D. PPP 모드
기준국 없이도 정밀 위치 얻는 기술. NGII NTRIP 못 쓰는 환경에서 대안.

## 부록: 한 페이지 요약

```
[전체 흐름]
모듈 켜기 → GPS만으로 위치 잡기 → NTRIP 접속 → RTCM 받기 시작
       → 모듈에 RTCM 흘려주기 → DGPS → Float → Fixed

[중요 숫자]
GGA 7번째 필드 = Fix Quality (0/1/2/4/5)
HDOP < 1.0 = 좋은 위성 배치
SNR > 40 = 양호한 신호 (다중경로 적음)

[흔한 함정]
- $GN vs $GP talker ID 차이
- 마운트포인트의 지원 위성 시스템
- MASK는 비활성화, UNMASK가 활성화
- 다중경로는 위상 측정의 적

[Fixed 못 얻는 99%의 이유]
안테나 위치 (실내, 창문, 건물 근처)
```
